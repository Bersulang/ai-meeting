# 架构治理一期灰度与回滚手册（PR-07）

## 1. 目标
- 覆盖 P0/P1 改造项上线流程：会话 owner 鉴权、WebSocket 鉴权、身份上下文收口、并发执行模型收口。
- 在出现兼容性或性能问题时，快速定位并回滚。

## 2. 发布前检查
- CI 必须通过：`backend-ci` 工作流 `clean test` 通过。
- 配置检查：
  - Sa-Token 配置正确，前端请求和 WebSocket 连接能携带 token。
  - Redis 可用（消息序号原子分配依赖 Redis）。
- 冒烟检查：
  - 普通聊天和 AI 面试聊天可正常发问、收流式结果。
  - WebSocket 语音转写在携带 token 时可连通，无 token 被拒绝。
  - 非 owner 访问会话历史返回拒绝。

## 3. 灰度步骤
1. 先灰度到测试环境，执行接口回归和并发压测（重点看消息序号是否递增无冲突）。
2. 生产环境先放小流量（建议 5%），持续观察 30 分钟：
   - HTTP 4xx/5xx 比例
   - WebSocket 握手失败率
   - Redis 命令耗时（`INCR`/`EVAL`）
3. 指标正常后扩大到 30%，再观察 30 分钟。
4. 全量发布后观察 2 小时，确认无新增告警。

## 4. 关键观测点
- 安全相关：
  - 非 owner 会话访问被拒绝（预期增加 4xx，但不应出现 5xx）。
  - WebSocket 无 token 或 userId 不匹配被拒绝。
- 并发相关：
  - 同一会话 `messageSeq` 连续递增，无重复。
  - 线程池活跃线程/队列长度可控，无大量拒绝执行。

## 5. 回滚策略
- 应用回滚：回滚到上一个稳定 tag/commit，重启服务。
- 数据层回滚：
  - 本期不涉及结构性 DB 变更，可直接应用级回滚。
  - Redis 的消息序号 key 可保留，不影响旧版本读取历史消息。
- 回滚触发条件（任一满足）：
  - 5xx 错误率显著升高且持续 10 分钟以上；
  - WebSocket 鉴权误杀导致核心功能不可用；
  - 聊天消息序号出现大面积重复或乱序。

## 6. 故障排查优先级
1. 先看鉴权链路：token 是否正确透传（HTTP header / WebSocket query）。
2. 再看会话 owner 校验：sessionId 对应 owner 是否一致。
3. 再看 Redis：是否可用、延迟是否异常。
4. 最后看线程池：是否饱和、是否有拒绝策略大量触发。

## 7. 责任分工建议
- 后端：发布、日志与指标监控、回滚执行。
- 前端：token 透传与错误提示兜底。
- 测试：非 owner 越权用例、WebSocket 非法连接用例、并发消息序号用例。
